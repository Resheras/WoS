Instruction architecture

Every layer has one job

The most common mistake is mixing routing logic, formatting rules, and capability descriptions across layers. Each layer must own exactly one concern.

Layer Owns Does not own
Orchestration instructions Tool selection logic, keyword signals, sequential patterns, scope boundaries, guardrails, audience definition Output formatting, citation style, number formatting
Response instructions Output format, tone, citation style, number formatting, data limitation transparency Routing logic, tool selection, guardrails
Tool descriptions What data the tool contains, when it is appropriate to use Formatting rules, routing hierarchies that span multiple tools
Red flag: "Do not speculate" appearing in both orchestration instructions AND a tool description. Consolidate all negative constraints into one authoritative list in orchestration.

Audience & domain context

Define the audience before anything else

Agents can't calibrate tone, depth, or relevance without knowing who is asking and what they value. This belongs at the top of orchestration instructions.

8

Audience definition

Name the primary users and what they value. E.g. "Sales reps and managers need fast, accurate, data-driven insights — not lengthy explanations."

9

Domain context

Name the business domain and data landscape so the agent understands its world. E.g. "This agent covers B2B deal metrics and sales conversation transcripts."

10

Numbers-in-context rule

Instruct the agent to never surface a bare metric. Every number must carry its denominator and sample context.

Example instruction

You are a sales intelligence agent for a B2B software company. Your primary users are sales representatives, sales managers, and revenue operations teams. They need fast, accurate, data-driven insights.

When querying metrics, always provide context. Don't say "win rate is 67%". Say "win rate for Enterprise Suite is 67% based on 2 wins out of 3 closed deals."

Behavioral controls

Teach the agent what not to do

Explicit hierarchies reduce tool selection errors by 40–60%. Signal words and sequential patterns are the most effective behavioral controls.

Explicit tool priority (check 12)

State the tool selection order clearly. "Always use Analyst first for quantitative questions. Only use Search after obtaining record identifiers from Analyst results."

Sequential multi-tool pattern (check 14)

For questions that filter unstructured data by structured criteria, enforce query-then-search:

"If asked about lost deals, first query metrics to find deals where win_status = false, then search conversations for those specific customers."

Empty-results handling (check 15)

If a structured query returns no data, report the gap honestly. Do not silently fall back to Search. Empty results signal data coverage limits — that is meaningful information.

Guardrails first (check 6)

Off-topic refusal, scope boundaries, and prediction blocks must appear at the top of orchestration instructions — not buried mid-document where they're easily missed.

Response quality

Format for humans, not for robots

Four response instruction checks that distinguish a working agent from a trustworthy one.

17

Lead with the direct answer

Structure: direct answer first, then supporting details. Prose over bullet points — conversational responses outperform lists for human readability and trust.

18

Concrete citations

"In the TechCorp discovery call..." not "according to the data." Specific citations are verifiable and expose hallucinations. Vague citations destroy trust.

19

Number formatting rules

Express deal values with K suffix ($90K not $90,000). Round percentages to whole numbers unless precision is explicitly needed. Format for humans, not spreadsheets.

20

Data limitation transparency

When synthesizing fewer than three data points, flag the sample size. "Based on two closed Enterprise deals" — not "Enterprise deals typically." Two deals are not a pattern.

Before: "Win rate is 67%."    After: "Win rate for Enterprise Suite is 67% based on 2 wins out of 3 closed deals — note this is a small sample."

Infrastructure & robustness

Two infrastructure checks that matter most

These are easy to miss and hard to debug once the agent is in production.

📌

Pin the orchestration model (check 21)

models.orchestration must be a specific model name — never "auto". The auto setting introduces non-determinism at the model selection level, compounding the natural non-determinism already present in LLM outputs. Pin to a specific model for reproducible behavior.

🎚

Check the max_results value on all search tools.

Iteration loop

Agents are never perfect on day one

Use traces to diagnose issues systematically. Each symptom maps to exactly one layer to fix.

1. Spot the symptom

Wrong tool used, poor format, speculation, vague citation, missing context.

→

2. Diagnose the layer

Wrong tool → orchestration. Bad format → response instructions. Speculation → boundary rules.

→

3. Write a specific fix

Describe a pattern, state a rationale, include an example. Vague instructions produce inconsistent behavior.

→

4. Verify via trace

Confirm planning span reflects the intended tool sequence. Check response span for format compliance.

→

5. Document the pattern

Build a library of fixes. New issues often match patterns you've solved before.

Critical

Silent failures, broken tool calls, no guardrails, typo in tool_resource name

High

No audience definition, missing citation format, no sequential multi-tool pattern

Nice-to-have

Keyword signal lists, number formatting rules, result count cap
